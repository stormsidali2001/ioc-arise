## Why IoC Arise Exists

The TypeScript ecosystem has two common approaches for dependency injection:

1. **Runtime DI frameworks** (InversifyJS, TSyringe) - require decorators throughout your codebase
2. **Manual wiring** - time-intensive and difficult to maintain at scale

**IoC Arise offers an alternative:** Automated code generation with compile-time analysis.

<Tabs>
<TabItem label="‚ùå Decorator-Based DI">

```typescript
// Domain code with framework decorators
@Injectable()
class UserService {
  constructor(
    @Inject('LOGGER_TOKEN') private logger: ILogger,
    @Inject('USER_REPO_TOKEN') private repo: IUserRepository
  ) {}
}

// Manual registration required
container.register('LOGGER_TOKEN', { useClass: ConsoleLogger });
container.register('USER_REPO_TOKEN', { useClass: UserRepository });
container.register(UserService, UserService);
```

**Tradeoffs:**
- Decorators create framework coupling
- Manual registration increases maintenance overhead
- Runtime reflection impacts performance
- String tokens lack compile-time validation
- Domain layer has infrastructure dependencies

</TabItem>
<TabItem label="‚úÖ IoC Arise Approach">

```typescript
// Pure TypeScript - no decorators, no framework coupling
class UserService implements IUserService {
  constructor(
    private logger: ILogger,
    private repo: IUserRepository
  ) {}
}

// That's it! Run one command:
// $ npx @notjustcoders/ioc-arise
```

**Generated Result:**

```typescript
// Auto-generated container with all classes registered
import { container } from './container.gen';

// Type-safe resolution with IntelliSense
const service = container.resolve('IUserService'); // UserService
```

**Key Advantages:**
- **No decorators** - domain code remains framework-agnostic  
- **Automated discovery** - CLI analyzes and registers all classes
- **Compile-time validation** - catch issues during build
- **Static analysis** - detects circular dependencies, duplicates, and collisions
- **Full type safety** - complete IntelliSense support
- **Modular output** - generates organized module files

</TabItem>
</Tabs>

:::tip[üí° How It Works]
IoC Arise uses **AST analysis** (powered by Rust) to understand your TypeScript code at compile-time. It analyzes interfaces, classes, and dependencies to generate **production-ready DI code** with better performance than runtime reflection.
:::

## Quick Start

<Steps>

1. **Install**

   ```bash
   npm install -D @notjustcoders/ioc-arise
   npm install @notjustcoders/di-container
   ```

2. **Write normal TypeScript** (no decorators!)

   ```typescript
   // Pure TypeScript with constructor injection
   export class UserService implements IUserService {
     constructor(private repo: IUserRepository) {}
   }
   ```

3. **Generate** (one command!)

   ```bash
   npx @notjustcoders/ioc-arise
   ```
   
   ‚ú® Generates:
   - `container.gen.ts` - Ready-to-use DI container
   - `modules/*.module.ts` - Organized by module  
   - `container.gen.d.ts` - Full TypeScript types

4. **Enjoy type-safe DI**

   ```typescript
   import { container } from './container.gen';
   
   const userService = container.resolve('IUserService');
   //    ^? IUserService - Full IntelliSense!
   ```

</Steps>

:::note[Quick Setup]
No configuration files required. No decorators to add. No manual registration. Simply write TypeScript classes and run the CLI to generate your DI container.

## How It Works

IoC Arise is a **two-package system** designed for maximum flexibility


1. **`@notjustcoders/ioc-arise`** (CLI Tool)
   - AST-powered code analyzer (Rust-backed)
   - Dependency graph builder
   - Circular dependency detector
   - Code generator

2. **`@notjustcoders/di-container`** (Runtime)
   - Lightweight DI container (~4KB)
   - Zero decorators, zero reflection
   - Can be used standalone without CLI
   - Framework-agnostic

## Head-to-Head Comparison

How does IoC Arise stack up against other approaches?

| Feature | IoC Arise üèÜ | Runtime DI Frameworks<br/><small>(TSyringe, InversifyJS, TypeDI)</small> | Manual DI<br/><small>(DIY)</small> |
|---------|-------------|--------------------------------------------------------------------------|-----------------------------------|
| **Zero Decorators** | ‚úÖ Pure TypeScript | ‚ùå `@Injectable()` required | ‚úÖ Yes |
| **Type-Safe Resolution** | ‚úÖ Full (generated `.d.ts`) | ‚ö†Ô∏è Partial (via decorators) | ‚ùå No autocomplete |
| **Compile-Time Errors** | ‚úÖ Catch before build | ‚ùå Fail at runtime | ‚ùå Fail at runtime |
| **Auto-Wiring** | ‚úÖ Fully automated | ‚ö†Ô∏è Manual decorators | ‚ùå 100% manual |
| **Circular Dep Detection** | ‚úÖ CLI + Runtime (both levels) | ‚ö†Ô∏è Runtime only | ‚ùå No detection |
| **Module Organization** | ‚úÖ Auto-generated files | ‚ö†Ô∏è Manual setup | ‚ö†Ô∏è Manual setup |
| **Name Collision Detection** | ‚úÖ Compile-time error | ‚ùå Manual resolution | ‚ùå Manual resolution |
| **Bundle Size** | ‚úÖ ~4KB runtime | ‚ö†Ô∏è 15-20KB | ‚úÖ 0KB (but labor cost) |
| **Framework Coupling** | ‚úÖ Zero (pure TypeScript) | ‚ùå Tight coupling | ‚úÖ Zero |
| **Learning Curve** | ‚úÖ Very low | ‚ö†Ô∏è Medium-High | ‚ö†Ô∏è High |
| **Refactoring Support** | ‚ö†Ô∏è Regenerate after refactor | ‚ö†Ô∏è String tokens break | ‚ö†Ô∏è Manual updates |
| **Production Ready** | ‚úÖ Day 1 | ‚úÖ Day 1 | ‚ö†Ô∏è Weeks of work |

:::tip[üí° Best of Both Worlds]
IoC Arise combines the **clean code** of manual DI with the **automation** of framework-based DI, while minimizing the tradeoffs of each approach.
:::
